1. Perubahan pada Formulir Pendaftaran & Tampilan Utama (Frontend)

* a. Tambahan Input Data: Pada formulir pendaftaran (di welcome.tsx bagian kontak), tambahkan kolom Jurusan (opsional) dan Keahlian/Keterampilan (teks area) tepat di atas kolom alamat lengkap.
* b. Informasi Keahlian: Data keahlian/keterampilan yang diisi oleh user ini harus dimunculkan pada setiap card pengajuan di dasbor Admin Verifikator (/verifikator/masuk) agar admin bisa melihat spesialisasi peserta.
* c. Kuota OPD (Inisiatif UX): Setiap card OPD di halaman utama wajib menampilkan informasi ketersediaan kuota magang. Admin OPD hanya bisa mengubah kuota OPD-nya sendiri, sedangkan Admin Verifikator memiliki akses untuk mengubah semua kuota.

2. Perombakan Alur Tugas Admin (Verifikator & OPD)

* a. Pemindahan Hak Penempatan: Kolom isian Divisi/Bidang, Pembimbing Lapangan, dan Penanggung Jawab dihapus dari pop-up "Tinjau Pengajuan" Admin Verifikator pada \Resources\js\Pages\verifikator\dashboard.tsx. Hak untuk mengisi ketiga detail tersebut dipindahkan ke pop-up "Detail Pengajuan" milik Admin OPD yang terletak pada \Resources\js\Pages\opd\dashboard.tsx.
* Tugas Baru Verifikator: Sebagai gantinya, pada pop-up tinjau pengajuan, Admin Verifikator kini bertugas memberikan "catatan khusus dari Admin Verifikator" yang nantinya akan dibaca oleh Admin OPD saat menerima lemparan data.
* Peringatan (Alert) untuk Admin OPD: Pada pop-up detail pengajuan Admin OPD, jika mereka memilih setuju, wajib ditambahkan teks alert/peringatan di bawah form yang berbunyi: "Notif ini akan dikirim ke peserta magang. Peserta akan datang berkunjung ke kantor setelah diterima pengajuan ini".
* Penyederhanaan Menu: Menu "Pengaturan" pada sidebar peserta magang dihilangkan sepenuhnya untuk mencegah admin kebingungan dalam mengubah data profil mereka sendiri.

3. Format Ulang Email Notifikasi Persetujuan

* Isi Email Lebih Detail: Jika Admin OPD menyetujui, email tidak sekadar berisi "pengajuan disetujui". Email harus memiliki header dengan teks besar berwarna biru bertuliskan "magang kota Madiun" dan status "disetujui".
* Kelengkapan Data: Email wajib menampilkan ulang detail magang peserta: asal instansi, durasi, serta rincian yang baru saja diisi Admin OPD (divisi, pembimbing lapangan, penanggung jawab, dan catatan).
* Instruksi Kedatangan: Email ditutup dengan instruksi yang sangat jelas, contohnya: "Silakan datang ke kantor OPD terkait, menemui bidang [Nama Bidang], lapangannya [Nama Pembimbing]".

4. Otomatisasi Status (Cron Job) & Penyelesaian Magang

* Filter Status Dasbor: Urutan filter status pada dasbor (terutama OPD) disesuaikan menjadi dari kiri ke kanan: Perlu Keputusan, Disetujui, Sedang Magang, Selesai Magang, Ditolak, Semua.
* Logika Cron Job (Scheduler): Anda diinstruksikan untuk menggunakan Cron Job di Laravel yang berjalan setiap hari (misal jam 1 pagi) untuk mengecek dan mengubah status secara otomatis:
* Jika status saat ini "Disetujui" dan tanggal hari ini sama dengan periode mulai magang, maka status otomatis berubah menjadi "Sedang Magang".
   * Jika status saat ini "Sedang Magang" dan tanggal hari ini sama dengan batas akhir magang, maka status otomatis berubah menjadi "Selesai Magang".
* 4 Aktor Status "Selesai": Status pengajuan berubah menjadi "Selesai" tidak hanya lewat sistem Cron Job. Status ini bisa dieksekusi oleh 4 pihak: 1) Sistem (Cron Job), 2) Admin Verifikator, 3) Admin OPD, dan 4) User (Anak Magang) saat mereka mengunggah laporan dan menekan tombol selesai.

5. Keamanan & Sistem Login Admin

* Single Session Browser: Keamanan diperketat di mana 1 browser tidak boleh bisa melakukan login untuk 2 akun berbeda secara bersamaan. Jika mencoba ditembus, sistem harus melempar Error 403 Forbidden.
* Akses Login Admin Dipisah: Untuk mempermudah, admin tidak lagi menggunakan alur OTP. Admin diberikan URL login khusus (misalnya /admin/login) yang langsung menggunakan form Username dan Password konvensional.
