# PRODUCT REQUIREMENTS DOCUMENT (PRD) - SISTEM E-MAGANG

## 1. Project Overview & Tech Stack
*   **Nama Proyek:** Sistem Manajemen E-Magang Terpusat.
*   **Tujuan:** Memfasilitasi pendaftaran, verifikasi, pelacakan, dan penyelesaian magang bagi mahasiswa ke berbagai OPD (Organisasi Perangkat Daerah) tanpa kerumitan registrasi akun manual.
*   **Tech Stack Utama:**
    *   **Backend & Frontend:** Framework **Laravel** (versi terbaru).
    *   **Database:** **PostgreSQL**.
    *   **Styling:** Bootstrap atau sejenisnya (bersih dan responsif).

## 2. User Roles & Permissions
Sistem ini memiliki 3 (tiga) peran utama:
1.  **User (Anak Magang):** Pengguna publik yang mengajukan form magang. Mengakses dasbor untuk melihat status, mengunggah laporan akhir, mengisi survei, dan mengunduh sertifikat.
2.  **Admin Verifikator:** Admin utama/tengah. Bertugas memverifikasi form masuk, meneruskan (*forward/lempar*) data ke Admin OPD spesifik beserta detail penempatan (bidang, pembimbing, penanggung jawab), mengelola FAQ, memvalidasi laporan akhir, dan mengunggah surat selesai magang.
3.  **Admin OPD:** Admin divisi akhir. Hanya menerima data yang diteruskan oleh Admin Verifikator dan bertugas menekan tombol **ACC (Terima)** atau **Tolak**.

## 3. Security Requirements (High Priority)
Keamanan harus diimplementasikan dengan sangat ketat sejak awal karena sistem tidak menggunakan registrasi akun manual:
*   **Vulnerability Prevention:** Wajib menerapkan anti-*Brute-force*, pencegahan *SQL Injection*, dan *XSS Injection*.
*   **Anti-Spam & Rate Limiting:** Batasi frekuensi pengisian form per hari berdasarkan kombinasi **IP Address + Email** atau **IP Address + Nomor WhatsApp**.
*   **Input Validation:** Validasi panjang karakter maksimal, dan pastikan field nomor WhatsApp hanya menerima input numerik.
*   **Captcha:** Wajib menggunakan *Captcha* interaktif pada form publik (model *slider/puzzle* geser seperti TikTok).

## 4. Authentication Logic (No-Password / OTP Flow)
*   **No Explicit Registration:** Pengguna baru TIDAK perlu menekan tombol "Daftar". Mereka langsung mengisi "Formulir Pengajuan Magang".
*   **Account Auto-Creation:** Setelah form dikirim pertama kali, sistem mengecek *database*. Jika Email/WA belum ada, sistem otomatis membuatkan akun *User* berdasarkan data form tersebut.
*   **Login via OTP:** 
    *   User mengakses halaman login dan HANYA memasukkan Email.
    *   Sistem mengecek email. Jika email tidak ada di tabel `users` (belum pernah isi form), tolak dengan pesan "Akun tidak terdaftar".
    *   Jika email ada, sistem *generate* **OTP (One Time Password)** dan mengirimkannya ke Email via SMTP.
    *   Sistem melakukan *hash* pada kode OTP tersebut dan melakukan *update* ke kolom `password` di tabel `users`.
    *   User memasukkan OTP untuk login. Jika cocok dengan *hash*, user masuk ke dasbor. Setiap kali login, alur OTP ini diulang sehingga fitur "Lupa Password" ditiadakan.
*   **Admin Access:** Admin hanya dapat melakukan *reset password* untuk user jika terjadi kendala, tetapi tidak bisa melihat *password* (karena ter-*hash*).

## 5. Core Features & User Workflows

### A. Public Landing Page
*   **Informasi:** Menampilkan **Alur Pendaftaran** dan ketersediaan/statistik (opsional).
*   **FAQ Dinamis:** Menampilkan *Frequently Asked Questions* yang kontennya bisa di-CRUD (tambah/edit/hapus) oleh Admin Verifikator melalui panel admin.

### B. Alur Pengajuan Magang (Formulir)
*   User mengisi form publik berisi: Tujuan Magang, Durasi (Berapa bulan, Tanggal Mulai-Selesai), Asal Kampus/Instansi, Nama Dosen Pembimbing, Nomor WhatsApp, dan Email.

### C. Alur Verifikasi (Admin Workflow)
1.  **Tahap 1:** Form masuk ke dasbor Admin Verifikator. Admin ini menambah detail (Diteruskan ke OPD mana, Bidang apa, Siapa pembimbing lapangan, Siapa penanggung jawab). Admin Verifikator bisa menolak langsung atau Meneruskan ke Admin OPD.
2.  **Tahap 2:** Admin OPD melihat data yang dilempar dan hanya memilih **ACC (Terima)** atau **Tolak**.
3.  **Notifikasi:** Jika Ditolak, kirim email penolakan. Jika ACC, kirim email persetujuan otomatis via SMTP. Email ACC **wajib melampirkan file PDF** (berisi surat rekomendasi/tanda terima resmi untuk dibawa saat magang, *layout* mirip *invoice*).

### D. User Dashboard & Ticket Tracking
*   Setelah login, User melihat tabel riwayat/tiket pengajuan magangnya.
*   Tiket memiliki status *real-time*: `Tahap Verifikator Admin`, `Tahap Admin OPD`, `Sedang Magang`, `Diajukan Proses Selesai`.

### E. Alur Selesai Magang & Sertifikat (Feedback Loop)
1.  **Upload Laporan:** Jika magang selesai, User menekan tombol di dasbor untuk **Upload Laporan Magang** (file) dan mencentang *checkbox* (konfirmasi "Saya sudah menyelesaikan magang").
2.  **Verifikasi Admin:** Laporan dikirim ke dasbor Admin Verifikator. Admin Verifikator mengecek laporan dan memberikan status ACC penyelesaian.
3.  **Upload Surat Selesai:** Admin Verifikator mengunggah dokumen **Surat Selesai Magang (Sertifikat)** ke dalam tiket user tersebut.
4.  **Survei Kepuasan (Wajib):** File surat selesai muncul di dasbor User. Namun, tombol *Download* di-*disable/lock*. User wajib mengisi form **Survei Kepuasan / Feedback** (Rating Bintang & Komentar) agar tombol *Download* terbuka.

Fowchart E-Magang

### 🟩 FASE 1: Pendaftaran (Sisi User)
**1. Kunjungan Website:** User (calon anak magang) mengunjungi halaman utama (*Landing Page*) yang berisi informasi alur pendaftaran dan FAQ.
**2. Pengisian Form:** User langsung mengisi **Formulir Pengajuan Magang** (tanpa klik tombol "Daftar Akun"). Data yang diisi meliputi: tujuan magang, asal, durasi, nama pembimbing, email, dan nomor WhatsApp.
**3. Pengecekan Database (Backend):** 
*   **Jika Email/WA belum ada:** Sistem **otomatis membuatkan akun** baru untuk user tersebut.
*   **Jika Email/WA sudah ada:** Sistem menggunakan akun yang sudah ada.

---

### 🟨 FASE 2: Verifikasi Bertingkat (Sisi Admin)
**4. Tahap 1 - Admin Verifikator:** 
*   Data pengajuan masuk ke panel **Admin Verifikator**.
*   Admin mengevaluasi kelengkapan form. 
*   Jika tidak valid, Admin Verifikator bisa **Langsung Tolak** -> *Kirim Email Penolakan*.
*   Jika valid, Admin melengkapi detail penempatan (memilih bidang, pembimbing lapangan, penanggung jawab), lalu **Meneruskan (Melempar)** pengajuan ke OPD tujuan.
**5. Tahap 2 - Admin OPD:**
*   Admin OPD terkait menerima data yang diteruskan.
*   Admin OPD hanya melakukan eksekusi akhir: **ACC (Terima)** atau **Tolak**.
*   *Jika Ditolak:* Sistem mengirimkan notifikasi penolakan via Email.
*   *Jika ACC:* Sistem otomatis mengirimkan notifikasi ke Email user beserta **lampiran dokumen PDF** (surat pengantar/rekomendasi magang). Status tiket berubah menjadi **"Sedang Proses Magang"**.

---

### 🟦 FASE 3: Akses Dasbor & Login OTP
**6. Permintaan Login:** User masuk ke halaman login dan **hanya memasukkan Email**.
**7. Validasi Akun:**
*   **Jika belum pernah isi form:** Sistem menolak akses dengan keterangan "Akun tidak terdaftar".
*   **Jika sudah pernah isi form:** Sistem men-*generate* kode **OTP (One Time Password)**, menyimpan *hash*-nya ke tabel *password*, dan mengirimkan OTP tersebut ke Email user.
**8. Masuk Dasbor:** User memasukkan OTP. Jika benar, user berhasil login dan dapat melihat tiket serta melacak status magangnya.

---

### 🟪 FASE 4: Penyelesaian Magang (Feedback Loop)
**9. Upload Laporan (User):** Setelah masa magang berakhir, user login kembali dan **mengunggah Laporan Magang** beserta mencentang konfirmasi bahwa mereka telah selesai magang.
**10. Verifikasi Laporan (Admin Verifikator):** Laporan tersebut masuk ke dasbor Admin Verifikator. Jika sudah sesuai, Admin **mengunggah Surat Selesai Magang (Sertifikat)** ke sistem (seperti *invoice* penyelesaian).
**11. Syarat Unduh (User):** 
*   Surat Selesai Magang muncul di dasbor User.
*   **Sistem mengunci tombol *Download***.
*   User **diwajibkan mengisi Survei Kepuasan (Feedback)** seperti memberikan *rating* bintang dan komentar.
**12. Magang Selesai:** Setelah form survei dikirim, tombol unduh akan terbuka, dan user dapat mengunduh Surat Selesai Magang mereka. Alur pengajuan dinyatakan selesai.

Semua alur di atas wajib berjalan di atas sistem keamanan yang ketat *(Anti-Brute Force, Anti-XSS, Anti-SQL Injection)* dengan pembatasan IP + Email dan perlindungan menggunakan *Captcha* yang digeser untuk mencegah bot/spam.
